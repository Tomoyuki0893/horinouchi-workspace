# 会計IF 必要ロジック抽出（画面起点）

BloomMaker 画面(TMH)から呼ばれている im-LogicDesigner フローを起点に、**会計IF に必要なロジックだけ**を抽出したもの。
全 **430 フロー**（エクスポート `im_logicdesigner-data.zip` / 696 定義・重複除外後 430）のうち、
画面から直接呼ばれるのは **30 フロー**（会計IF カテゴリ 29 + 社内アプリケーション 1）。

- 解析対象: `intra-mart/bloomMaker/TMH`（画面14）, `intra-mart/logic-designer/TMH`（ルート286・フロー本体）
- 呼び出しアクション: `SendRequestImLogicFlowRouteActionItem`（全37呼び出し / ユニークルート30）
- 全フロー `transaction = true`、SQL は 2-way SQL（`/*param*/'default'` 形式のバインド）で SQL インジェクション耐性あり

## 2. CRUD 検証所見（SQL 本体まで確認）

### 🔴 要修正（確定バグ）
1. **仕掛品出庫「更新」フローの SQL が構文エラー**（`仕掛品出庫入力データ更新`）
   - `SET` 句で **カンマ抜け 2 箇所**（`recorded_date=…''` の後、`issue_date=…''` の後）
   - さらに **`WHERE` 直前に余分なカンマ**（`issue_destination=…0,` の直後）
   - → このままでは UPDATE が **実行時 SQL 構文エラー**。
   - 加えて、更新（状態変更）を **HTTP GET** で公開している（他画面の更新は PUT `/edit`）。GET はキャッシュ/リトライで副作用の恐れ。**PUT/POST へ変更推奨**。

### 🟠 要確認（不具合の可能性が高い）
2. **TMC買掛金不照合追加：チェック工程が「売掛金」フローを呼んでいる**
   - 買掛金の画面なのに、check が `TMC**売掛金**不照合追加入力データチェック`（別カテゴリ [会計IF / TMC売掛金…]）を参照。**取り違え（コピペ）の疑い**。
   - 登録ルートが `01JZ9F9D…/check` という名前だが実体は **INSERT 登録**フロー。ルート名と処理内容の不一致（紛らわしい）。
   - 画面側の入力検証（`validationRules`）が **0 件**。他の追加系画面（11〜15件）と比べ手薄。

### 🟡 要検討（一貫性・安全性）
3. **資材差額調整の削除 WHERE が単一キー**：`DELETE … WHERE issue_serial_number = …` のみ。
   他の削除（特定工数・加工不良）は `company_identifier` 等を含む複合キー。
   `issue_serial_number` が会社横断で一意でない場合、**会社跨ぎ削除**の恐れ → スコープ確認。
4. **部品仕入単価のルート名が単複混在**：`..._price/check`・`..._price/new`（単数）と `..._prices`・`..._prices/edit`（複数）。
   同一テーブル `tmh_acif_part_purchase_price` を指すため機能はするが、命名の統一が望ましい。
5. **ルート命名が 2 系統混在**：ULID 系（`01J0Z…/getList`）と名前系（`tmh_*/check|new|edit|delete`）。
   後者（模範3画面）に統一するとメンテ性が向上。

### ✅ 良い点
- **マスタメンテ3画面（資材差額調整・特定工数・加工不良）は CRUD が完備**：
  search → check（重複/整合チェック, SELECT複数）→ new（INSERT）→ edit（UPDATE）→ delete（DELETE）が、
  確認ダイアログ（各3件）・エラーダイアログ・入力検証とともに一貫実装。**これが模範パターン**。
- 全フロー `transaction = true`。UPDATE/DELETE は（資材差額deleteのスコープ点を除き）WHERE 付き。
- SQL は 2-way SQL バインドで、値の文字列連結による注入は見られない。

## 4. 検証範囲・前提

- 本抽出は **画面(BloomMaker)→ルート→フロー→SQL** の静的追跡による。
- サブフロー参照（`/check` が内部で呼ぶ検証サブフロー等）は 1 段は追跡済み。多段の完全展開は未実施。
- 実行時のデータマッピング（画面変数 ⇔ フロー入出力）の整合、権限（`secured=false`・`useSecureToken=0`）の妥当性は別途要確認。
