# 会計IF パラメータ受け渡し検証(BloomMaker ⇔ LogicDesigner)

BloomMaker 画面が検索/登録/更新でフローを呼ぶ際の、**①BloomMaker送信項目 ②フロー入力定義(REST契約) ③実SQLバインド** の三層突合結果。

## 検索(search)

**14画面中12〜13は完全一致**(送信項目=入力定義、過不足なし)。問題は1画面。

### 🔴 部品仕入単価(検索)— 入力定義が姉妹「部品売上単価」の使い回し
| 検索項目 | ①送信 | ②入力定義 | ③実SQL | 判定 |
|---|:-:|:-:|:-:|---|
| parts_number / management_number / 適用開始・終了日 | ○ | ○ | ○ | ✅効く |
| 仕入先 supplier_identifier | ○ | **✗** | ○ | 🔴契約に無く渡らない疑い |
| 出荷先 issue_destination | ○ | **✗** | ○ | 🔴同上 |
| 単価 unit_price_from/to | ○ | **✗** | ○ | 🔴同上 |
| 設定日 configuration_date | ○ | **✗** | ○ | 🔴同上 |
| trading_partner_identifier / combined_supply_type / external_sale_type | ✗ | ○ | ✗ | ⚫死んだ定義(売上流用) |

→ 品番・管理番号・適用日だけが効き、**仕入先/出荷先/単価/設定日の絞り込みが効かない**可能性大。
入力定義を実SQL(supplier_identifier 等)に合わせて修正が必要。

### 🟡 特調たな卸単価未決(検索)
`company_identifier` が入力定義にあるがBloomMakerは未送信。会社IDをサーバ側解決なら可。SQLのWHEREで使うなら要確認。

## 登録・更新(new / edit)

**登録・更新フロー本体はおおむね正常**(完全一致、差は `flag` 等の制御項目が余分に送られる程度で無害)。
- 完全一致: TMC買掛金**登録**、資材差額 登録/更新、特定工数 登録/更新、加工不良 登録、仕掛品出庫 更新、部品仕入単価 登録/更新 ほか。

### 🔴 TMC買掛金不照合追加(チェック)— 売掛金フロー＋契約不一致
買掛金画面のチェックが **「TMC売掛金不照合追加入力データチェック」**(別ドメイン)を呼び、
入力定義の項目名も別体系(`partner_id / shukka_date / tmckenshu_date / correctreason_type / recording_date / gogutihokyu_type` …)。
BloomMakerは英語名(`trading_partner_identifier / issuance_date / tmc_delivery_date` …)で送るため、
**`delivery_note_number` と `management_number` 以外ほぼ全項目が噛み合わず、チェックにデータが渡らない**。
※登録フロー自体は正常なので、**チェックだけ空振り**。呼び先を買掛金用チェックに差し替え要。

### 軽微(設計内 or 無害)
- 加工不良チェック: 数量3項目(previous_month_qty/teiseiqty/tougetuqty)を受け取らない=キー存在チェックのみと推測。
- 加工不良更新: item_name / previous_month_qty が送信され入力定義に無い=無視。
- 各登録で `flag`(チェック/モード切替用)が余剰送信=無害。
