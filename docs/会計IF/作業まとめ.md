# 会計IF 作業まとめ（index・状態・決定事項）

TMH 会計IF（BloomMaker 14画面 + LogicDesigner フロー）のレビューと改修の全体像。
迷ったらまずこのファイル。

## 1. 全体像・目的
- BloomMaker の会計IF画面が使う LogicDesigner フローを対象に、①不具合の洗い出しと修正、
  ②登録/更新/削除が無い画面への新規CRUD作成、③配布用の絞り込み（トリム）を行う。
- 対象は **会計IF（`tmh_acif_*`）のみ**。他モジュール・他資材は触らない。

## 2. 決定事項
- **日付フォーマットは `yyyyMMdd` に標準化**（2026-07-19 決定）。
  - 「値（DB・フロー入出力・画面の送信値）」= **`yyyyMMdd`**。
  - 「表示」= `yyyy/MM/dd`（画面ウィジェットの表示形式でのみ整形）。
  - 影響: 値が全層 `yyyyMMdd` で揃うため、**WHERE の `REPLACE` も SELECT の `SUBSTRING` 変換も原則不要**になる。
    → 「フローごとに REPLACE/SUBSTRING を足す」方針は**この標準化で置き換え**。
  - 当面の防御策として WHERE の `REPLACE(param,'/','')` は入れても無害（スラッシュ無しなら無変換）。
- トリム配布は「画面が呼ぶ分だけ（29フロー）」。会計IF外の `【受注】検索` は除外（既存前提）。
- 全テナントダンプ（41MB）は Git 管理外（`.gitignore`）。配布はトリム版を使用。

## 6. 進め方（日付標準化 後の流れ）
1. 画面（BloomMaker）の日付「値」を `yyyyMMdd` に統一（表示は `yyyy/MM/dd`）。
2. それに伴い、フローの SELECT の日付 `SUBSTRING` 変換は表示専用に整理、WHERE は素の比較でOK
   （防御で REPLACE を残すのは可）。
3. B〜D の不具合修正 → 再エクスポート → 私が突合。
4. テーブル情報が届いたら新規CRUDを設計書ベースで作成（Designerでクローン）。
5. 最終的に修正版で再トリム → 検証テナントでテストインポート。
