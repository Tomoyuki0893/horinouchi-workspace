# 運用ルール（ソロ）

少人数・自分中心の運用を想定した、最小限で崩れないルール。

## ブランチ

| ブランチ | 役割 |
|---|---|
| `main` | 常に「動く・本番相当」の状態を保つ |
| `feature/*` | まとまった変更・実験用。完了したら `main` へマージして削除 |

- 小さな修正は `main` に直接コミットしてOK。
- 影響が大きい／試行錯誤する変更だけ `feature/*` を切る。

```bash
git switch -c feature/im-workflow-申請
# ...作業・コミット...
git switch main
git merge feature/im-workflow-申請
git branch -d feature/im-workflow-申請
```

## コミットメッセージ

- 1行目に「何を」を簡潔に（日本語可）。必要なら本文で「なぜ」を補足。
- 対象領域を接頭辞で示すと後から追いやすい:
  - `talon: 送り状取得のリトライ処理を追加`
  - `im-bis: 経費申請フォームの定義を更新`
  - `docs: 運用ルールを追記`

## リリース／区切りの記録

本番反映や大きな区切りでは注釈付きタグを付ける。

```bash
git tag -a v2026.07.18 -m "ささえあ 送り状連携 リリース"
git push --tags
```

## Intra-mart 定義ファイルの扱い

- エクスポートした定義（XML/JSON）は該当フォルダ（`intra-mart/bis` など）へ配置。
- 機械生成で差分が読みにくい定義は、必要に応じて `.gitattributes` に
  `パス -diff` を追記してレビュー対象から外す。
- テナントエクスポートの **zip をそのまま管理したくなった場合** は、
  サイズが大きくなるため Git LFS の導入を検討する（その時点で相談）。

## 秘密情報

- DB接続・SMTP・APIキー等はコミットしない。実行環境／マスタテーブルから取得する。
- 誤ってコミットした場合は履歴からの削除が必要になるため、`git push` 前に
  `git status` / `git diff --staged` で確認する習慣をつける。
