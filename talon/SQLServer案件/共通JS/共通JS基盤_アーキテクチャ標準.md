# TALON 共通JS基盤 アーキテクチャ標準

> 本ドキュメントは SQLServer案件における共通JS（サーバーサイド／クライアントサイド）の **標準構成** を定めるものです。
> 以後の共通処理の追加・改修は、原則すべて本ドキュメントを基準とします。

- 対象環境: TALON JavaScriptエンジン（Nashorn / **ES5準拠**）、SQLServer
- ステータス: ドラフト（レビュー依頼中）
- 最終更新: 2026-06-27

---

## 1. 目的

機能ごとに各自が回避策やユーティリティを書き散らす状態をやめ、共通処理を **層（レイヤー）** で整理する。これにより、

- REST連携・ログ・エラー処理などの **書き方を統一** する
- Nashorn(ES5)・SQLServer固有の制約を **下位層に閉じ込める**
- 後からの差し替え・テストを容易にする

---

## 2. 全体構成（5層）

上位層は **直下の層のみ** に依存する。下位層は上位層を一切参照しない。

```
┌─────────────────────────────────────────────────────────────┐
│ ① 業務機能層（呼び出し側）                                    │
│   画面・イベントJS ／ バッチ処理 ／ 外部システム連携          │
└───────────────┬─────────────────────────────────────────────┘
                │ 依存
┌───────────────▼─────────────────────────────────────────────┐
│ ② 業務共通層                                                  │
│   ★REST共通基盤 ／ DB・SQL補助 ／ バリデーション ／ 帳票・CSV・メール │
└───────────────┬─────────────────────────────────────────────┘
                │ 依存
┌───────────────▼─────────────────────────────────────────────┐
│ ③ 横断基盤層                                                  │
│   ログ共通 ／ エラーハンドリング ／ 定数・メッセージ ／ 設定読込 │
└───────────────┬─────────────────────────────────────────────┘
                │ 依存
┌───────────────▼─────────────────────────────────────────────┐
│ ④ コアユーティリティ層（土台）                                │
│   ★JS↔Java変換 ／ 日付・文字列・数値 ／ JSONヘルパー          │
└───────────────┬─────────────────────────────────────────────┘
                │ 実行基盤
┌───────────────▼─────────────────────────────────────────────┐
│  TALON JavaScriptエンジン ランタイム（Nashorn / ES5 / SQLServer）│
└─────────────────────────────────────────────────────────────┘

★ = 最初に整備する核（REST共通基盤・JS↔Java変換）
```

---

## 3. 依存ルール（3原則）

1. **下方向のみ依存**：①→②→③→④の順にのみ依存する。逆参照・層飛ばし（①→④直接など）は原則禁止。
2. **循環参照の禁止**：同一層内も含め、相互参照を作らない。共通で必要になったら下位層へ降ろす。
3. **固有制約は下位に閉じ込める**：ES5制約（`.map`不可など）は④に、SQLServer方言（ページング等）は②のDB補助に隔離し、業務側に漏らさない。

## 4. 各層の責務と公開関数

> 関数名・引数は暫定案。レビューで確定する。

### ④ コアユーティリティ層（土台）

| モジュール | 責務 | 公開関数（案） |
|---|---|---|
| **JS↔Java変換** ★ | Nashornの Java型配列／JS配列の相互変換、ES5で使えない反復処理の代替 | `toJsArray(javaArr)` / `toJavaArray(jsArr)` / `each(arr, fn)` / `mapTo(arr, fn)` |
| 日付・文字列・数値 | 日付整形（和暦・元号）、ゼロ埋め、カンマ編集、丸め | `formatDate(d, fmt)` / `toWareki(d)` / `zeroPad(v, len)` / `formatNumber(v)` |
| JSONヘルパー | 安全なJSONパース／文字列化、null安全アクセス | `parseJson(str)` / `stringify(obj)` / `getPath(obj, path, def)` |

### ③ 横断基盤層

| モジュール | 責務 | 公開関数（案） |
|---|---|---|
| ログ共通 | レベル制御（DEBUG/INFO/WARN/ERROR）、処理ID・ユーザID付与、開始/終了/例外ログ | `log.info(msg)` / `log.error(msg, e)` / `log.start(name)` / `log.end(name)` |
| エラーハンドリング | 例外の集約ラップ、業務エラーコード体系、標準エラーレスポンス整形 | `newError(code, msg)` / `wrap(e)` / `toErrorResponse(e)` |
| 定数・メッセージ | メッセージID→文言の解決、共通定数の集約 | `msg(id, args)` / `CONST.XXX` |
| 設定読込 | プロパティ／環境設定値の取得 | `config.get(key, def)` |

### ② 業務共通層

| モジュール | 責務 | 公開関数（案） |
|---|---|---|
| **REST共通基盤** ★ | HTTPクライアント、トークン認証（Ver6.2.3〜）の取得・保持・再取得、共通ヘッダ、タイムアウト／リトライ、レスポンス→業務エラーコード変換 | `callApi(method, path, body, opts)` / `get(path)` / `post(path, body)` / `getToken()` |
| DB・SQL補助 | ページング（SQLServer: OFFSET/FETCH 等）、IN句の動的生成、件数取得＋データ取得のセット、トランザクション境界の定型化 | `paging(sql, page, size)` / `buildInClause(list)` / `selectCount(sql)` |
| バリデーション | 必須・桁数・型、郵便番号／電話／メール／日付形式、全角半角判定 | `required(v)` / `isMail(v)` / `isZip(v)` / `isDate(v)` |
| 帳票・CSV・メール | 帳票（IOWebDOC連携）・CSV出力・メール送信の共通化 | `outputCsv(rows, header)` / `sendMail(to, subj, body)` |

### ① 業務機能層（呼び出し側）

各機能（画面JS・バッチ・外部連携）。共通処理は **必ず②③④経由** で利用し、機能内に共通ロジックを直書きしない。

## 5. 命名・配置規約（案）

- ファイル名: `層プレフィックス_モジュール名.js`（例: `core_javaConv.js` / `base_log.js` / `biz_restClient.js`）
- 名前空間: グローバル汚染を避け、`COM.core` / `COM.base` / `COM.biz` 配下に関数を登録する
- 1モジュール1ファイル。公開関数はファイル末尾でまとめてエクスポート

> ※ プレフィックス・名前空間ルールはレビューで確定する。

---

## 6. Nashorn / ES5 制約（共通ルール）

サーバーサイドJSは以下を厳守する。

- `const` / `let` は使わず **`var`** を使用
- アロー関数 `=>` は **使用不可**
- 配列の `.map()` / `.filter()` / `.forEach()` は Java型配列のため **直接使用不可** → ④の `each()` / `mapTo()` を使う
- テンプレートリテラル（バッククォート）は不可 → 文字列連結
- API仕様は TALON JavaScriptエンジンAPIマニュアルに準拠

---

## 7. 構築順序（ロードマップ）

土台から積み上げる。`★` が最優先。

1. **④ JS↔Java変換** ★ — 以降すべての反復処理の前提
2. **③ ログ共通 ＋ エラーハンドリング** — REST基盤の中身を薄くするため先に整備
3. **② REST共通基盤** ★ — 上記が揃ってから着手
4. ④ 日付・文字列・数値 / JSONヘルパー（随時）
5. ② DB・SQL補助 / バリデーション（随時）
6. ② 帳票・CSV・メール、③ 定数・メッセージ・設定読込（必要に応じて）
